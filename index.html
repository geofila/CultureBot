Why CultureBot

Cultural inquiry doesn't fit a search box

The mismatch

Digital heritage collections bring together objects, images, archival descriptions, and institutional metadata from many providers — formalized through models like the Europeana Data Model (EDM) and CIDOC CRM. Yet most users still access this richness through keyword search and faceted filters, which force them to think like the catalogue rather than ask the relational, interpretive questions they actually have: which objects relate to a place and period, how artefacts differ across providers, what themes emerge across a set of records.

Cultural records also carry evidential weight and provenance — dates are approximate, attributions uncertain, places historically ambiguous. A fluent but unsupported generated answer can create misleading historical connections and sever the link between users and the institutions that hold the originals.

The approach

CultureBot is a reusable framework for natural-language exploration of cultural heritage data, built around three commitments:

  • Structured + textual retrieval, unified. Queries about places, periods, object types, providers, or images are handled through the same pipeline.
  • Generation as navigation, not replacement. Answers cite supporting records and guide users back to the original repository pages.
  • Repository-independent design. Built for any EDM-style collection, not tied to one institution or dataset.
Architecture

From a question to a grounded, sourced answer

Given a natural-language query, CultureBot grounds it against a cultural knowledge graph, retrieves records through structured metadata relations and hybrid textual search, fuses the two, and generates a source-linked answer — never sending the raw query straight to a language model.

🗂️ Cultural Data EDM-style collection
→
🕸️ Knowledge-Graph Retrieval Grounded entities → validated Cypher → structured records
🔎 Textual Retrieval Hybrid lexical (BM25) + semantic (FAISS) search
→
🧬 Retrieval Fusion Merge, dedupe, attach metadata + text
→
📜 Source-Linked Answer Cited records, abstains when unsupported

1 · Query Grounding

User terms (e.g. “Thessaloniki area”) are fuzzy-matched against the graph and external vocabularies to surface candidate places, providers, and museums before any query is generated — so the model grounds itself in entities that actually exist.

2 · Validated Cypher Generation

An LLM translates the grounded query into read-only Cypher. Every query is checked for syntax, blocked write/admin operations, and verified against Neo4j with EXPLAIN before it is ever executed.

3 · Hybrid Textual Scoring

Lexical and dense retrieval are combined into a single relevance score:

s(q, r) = α · slex(q, r) + (1 − α) · ssem(q, r)

Lexical retrieval anchors names, dates, and identifiers; dense retrieval captures paraphrase and thematic similarity.

🛡️

Generated graph queries are conservative by construction: they must start with MATCH/OPTIONAL MATCH, contain a RETURN clause, and pass a read-only safety check that rejects CREATE, MERGE, DELETE, SET, DROP, LOAD CSV, and write-oriented APOC procedures — a query that fails validation is never run.

Instance

Running on SearchCulture

CultureBot is repository-independent by design, and is currently instantiated on SearchCulture data covering the movable monuments of the Directorate of Mobile Monuments, Hellenic Ministry of Culture.

Graph layer

Structured metadata projected into a Neo4j property graph — Object, Place, TimeSpan, Concept, Provider, Aggregation, Image, Rights, SourceURI — enriched with external vocabularies and place hierarchies.

Textual layer

A hybrid FAISS + BM25 index over record renderings, deduplicated and combined with reciprocal rank fusion. Embeddings: text-embedding-3-large.

Serving layer

Deployed as an OpenWebUI pipeline with configurable retrieval modes — kg, rag, and the default hybrid — and runtime valves for model, context size, and retrieval depth.

Evaluation design

Tested by the people who use these collections

Six domain experts — archaeologists and cultural heritage researchers — defined 12 realistic personas and authored 52 evaluation queries spanning factual lookup, spatial exploration, thematic comparison, object selection, image-oriented requests, and multi-step reasoning.

Per-query, per-system

Each of the 52 queries was answered by all four systems and rated by an expert against the originating persona — 208 expert answer evaluations in total. A parallel query-level assessment captured how hard each query would be to answer through manual SearchCulture search, and how long that would take.
